Leveraging docker's volume

I have been using docker for a while, but the concept of volume in docker has confused me for a long time. So I decided to write a blog on how to use it.

Sharing data between containers

docker run -ti -v “$PWD”:/dir/in/container image-name /bin/bash

-t means start a tty(terminal console from my point of view)
-i means interactive mode
By using this command, we tell docker to start a container binding the current directory in the host with the directory in the container.

To check the info of the container, we basically type docker inspect . Further more, to check the mounting status(how the volume is mounted between host and container), we can use the following command.

Inspired by this command, I think we can provide other key-word in curly brace to check other info since the docker inspect is simply returning a json format data.

Sharing data between different containers

docker run -ti -v /bin/bash

By ignoring the host directory, we are letting docker to assign the mounting /var/lib/docker/volumes/ as the host volume to the directory that we specified in the command. We can edit the volume in the host directory. Later on, even if we stop the container, once we start it again with docker start <container-id>, the edited content will get updated in the container.

In order to let other container visit this volume, we use the following command:

docker run -ti –volumes-from /bin/bash

2 things to notice:

  1. even if the container is not running, we can still mount it.
  2. we need to do docker rm -v container to remove the persisted volumes too, otherwise there will exist many unused volumes.
