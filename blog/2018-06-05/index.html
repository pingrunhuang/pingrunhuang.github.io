postgresql command note
ad
Jun 5, 2018
2 minutes read

Initial

  • login to postgresql: psql -U demo -p 15432 -d database -h host
  • list databases in postgresql: psql -l
  • backup database: PGPASSWORD="password" pg_dump -h host -p port -U username database > file.sql or PGPASSWORD="password" pg_dump -h host -p port -U username -t tablename database > file.sql for only one table.
  • recommended way to dumnp database: pg_dump -Fc mydb > db.dump
  • restore database: psql -U postgres database < file.sql
  • dump sql create table: pg_dump -p <port> -U <user> -h <host-ip> -st tablename dbname
  • create database: createdb <dbname>

After login

  • list tables in databases: # \d.
  • list tables in databases: \dt for listing tables in current database, programmatically SELECT * FROM pg_catalog.pg_tables;.
  • describe table in databases: # \d tablename or # \d+ tablename for further info.
  • switch database: \c dbname
  • list all the databases: \l
  • create database: create database dbname;
  • create table: ``` CREATE TABLE films ( code char(5) CONSTRAINT firstkey PRIMARY KEY, title varchar(40) NOT NULL, did integer NOT NULL, date_prod date, kind varchar(10), len interval hour to minute );

```

schema

  • analyze query efficiency: EXPLAIN ANALYZE <query statement>;
  • change to different schema: SET search_path = <schema_name>, this allow to change to different schema at run time
  • change the default schema for login: ALTER USER postgres SET search_path = <schema1>,<schema2>...
  • create schema: CREATE SCHEMA <schema_name>
  • move table from one schema to another: ALTER TABLE <tablename> SET SCHEMA <schema_name>

CRUD

  • insert: insert into tablename(c1,c2...) values (v1,v2...);
  • update table table column type: alter table <table name> alter column <column name> type <new_type>. Notice that for other databases, the syntax for alter table is alter table <table name> modify <column name> <new type>.

issues

  1. The reason why select count(*) is slow is related to the MVCC implementation in PostgreSQL. The fact that multiple transactions can see different states of the data means that there can be no straightforward way for “COUNT(*)” to summarize data across the whole table; PostgreSQL must walk through all rows, in some sense. This normally results in a sequential scan reading information about every row in the table. EXPLAIN ANALYZE reveals what’s going on
