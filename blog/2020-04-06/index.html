Again, these patterns are not supposed to cover all of the pattern that exists in the tech world. Patterns like Singleton, Factory, Builder .etc might not show up here since they are quite common in most code bases. I would only note those that I think might be helpful for my or your future career.

Prototype pattern

A creation pattern which allow faster creation of object with a simple clone method.

Adapter pattern

A structural pattern for modifying the existing class(either in place or by extending from it) to adapt to other classes.

Flyweight pattern

In such pattern, variables in a class are divided into 2 types, intrinsic(variables that are unique to certain object) and extrinsic(variables that are shared outside object).
A structural pattern that suggests object only contains intrinsic variables and should not be able to modified outside. For me, this pattern is more of combining singleton with proper separation. Why say so? Check out the following scenario.

class Stuff:
    def __init__(self, name, birth, gender, age, dep, pos):
        self.name = name
        self.birth = birth
        self.gender = gender
        self.age = age

class Company:
    def __init__(self, stuffs=None):
        self.stuffs = [] if not stuffs else stuffs

In the situation where the company is just a startup, we might not come across of memory issue, but with the company grow into a global company, hundreds of thousands of people might become stuffs of this successful company. We might run out of memory very quickly. The thing is we could extract the common part of the Stuff like gender, age, department and position into another class called StuffType and make sure only few of the instances get instantiated. class Stuff:
    def __init__(self, name, birth, _type):
        # name and birth are called 

        self.name = name  
        self.birth = birth
        self.stuff_type = _type

# this is called the flyweight: contains fields that could be referenced by many objects

class StuffType:
    def __init__(self, age, gender):
        self.age = age
        self.gender = gender

# I was thinking of using decorator here to keep the stuff_type attribute unique just like the singleton does

# Yes we can do that, but it is not that easy to understand though

class StuffTypeFactory:
    type_map = {}
    @staticmethod
    def gen_type(age, gender):
        key = (age, gender)
        if not type_map.get(key):
            StuffTypeFactory[key] = StuffType(age, gender)
        return type_map.get(key)
